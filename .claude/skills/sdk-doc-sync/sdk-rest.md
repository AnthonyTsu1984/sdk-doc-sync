# RESTful API Reference (Milvus server)

**Source repo:** `repos/milvus` (git worktree of `repos/milvus-sdk-go`, same milvus monorepo)
**Source files:** `internal/distributed/proxy/httpserver/`

| File | Role |
|------|------|
| `constant.go` | Route category prefixes (`/collections/`, `/entities/`, etc.) and action strings |
| `handler_v2.go` | All v2 route registrations via `RegisterRoutesToV2()` |
| `handler_v1.go` | v1 legacy routes |
| `request_v2.go` | Request struct definitions for v2 endpoints |

**Target files** (edited directly — no Feishu drive/bitable involved):

| File | Contents |
|------|----------|
| `specs/openapi-milvus.json` | Milvus server paths (`/v2/vectordb/*`, `/v1/vector/*`) + component schemas |
| `specs/openapi-cloud.json` | Zilliz Cloud paths — **do not touch when syncing from server source** |
| `specs/openapi.json` | Combined file — regenerated by merge script, not edited directly |

**Tag format:** `v*` (main milvus server releases, e.g., `v2.6.3`) — different from Go SDK `client/v*`.

**URL structure:** All v2 routes follow `/v2/vectordb/{category}/{action}`. All routes are `POST`.

## Workflow

**Phase 1: SCAN** — Read `handler_v2.go` for new/changed routes:

```bash
grep "router\.POST" repos/milvus/internal/distributed/proxy/httpserver/handler_v2.go
cat repos/milvus/internal/distributed/proxy/httpserver/constant.go
grep -A 20 "type FooReq struct" repos/milvus/internal/distributed/proxy/httpserver/request_v2.go
```

**Phase 2: DIFF** — Compare routes in source vs paths in `specs/openapi-milvus.json`.

**Phase 3: APPROVE** — Present new/changed/removed endpoints to user.

**Phase 4: EXECUTE** — Edit `specs/openapi-milvus.json` directly:
- Add new path objects following existing schema conventions
- Update request/response bodies to match new struct fields
- **Never edit `specs/openapi.json` directly** — regenerated by merge script

**Phase 5: MERGE:**

```bash
node scripts/merge-openapi.js
```

## OpenAPI Path Object Conventions

```json
"/v2/vectordb/collections/create": {
  "post": {
    "summary": "Creates a collection",
    "description": "...",
    "tags": ["Collection Operations (V2)"],
    "parameters": [{ "$ref": "#/components/parameters/AuthorizationHeader" }],
    "requestBody": {
      "content": {
        "application/json": {
          "schema": { "$ref": "#/components/schemas/CreateCollectionRequest" }
        }
      }
    },
    "responses": { "200": { "description": "...", "content": { "..." } } },
    "x-i18n": { "zh-CN": { "summary": "..." } }
  }
}
```

Key conventions:
- Always include `"$ref": "#/components/parameters/AuthorizationHeader"` in `parameters`
- Tag names follow pattern `{Category} Operations (V2)` — check existing tags before adding new ones
- Include `x-i18n` with `zh-CN` translations for `summary` and field descriptions
- Cloud paths (`/v2/clusters`, `/v2/alertRules`, etc.) are in `openapi-cloud.json` — never mix

## Scripts (in `scripts/` root — not moved)

- `scripts/edit-openapi.js` — CLI for reading/editing OpenAPI spec
- `scripts/merge-openapi.js` — merge milvus + cloud files into combined spec
- `scripts/split-openapi.js` — split combined spec into milvus + cloud files
