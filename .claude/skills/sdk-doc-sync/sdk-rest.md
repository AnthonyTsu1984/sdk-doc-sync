## OpenAPI Path Object Conventions

```json
"/v2/vectordb/collections/create": {
  "post": {
    "summary": "Creates a collection",
    "description": "...",
    "tags": ["Collection Operations (V2)"],
    "parameters": [{ "$ref": "#/components/parameters/AuthorizationHeader" }],
    "requestBody": {
      "content": {
        "application/json": {
          "schema": { "$ref": "#/components/schemas/CreateCollectionRequest" }
        }
      }
    },
    "responses": { "200": { "description": "...", "content": { "..." } } },
    "x-i18n": { "zh-CN": { "summary": "..." } }
  }
}
```

Key conventions:
- Always include `"$ref": "#/components/parameters/AuthorizationHeader"` in `parameters`
- Tag names follow pattern `{Category} Operations (V2)` — check existing tags before adding new ones
- Include `x-i18n` with `zh-CN` translations for `summary` and field descriptions
- Cloud paths (`/v2/clusters`, `/v2/alertRules`, etc.) live in the zdoc Cloud fragments (`15-*.json`–`30-*.json`) for public docs. Use `openapi-cloud.json` only as a baseline/reference when explicitly syncing skill-local specs.

## Zilliz Cloud API (openapi-cloud.json)

**Source repo:** `repos/zilliz-cloud` (Java, Spring Boot)
**Source files:** `vdc/global/cloud-control-api/src/main/java/com/zilliz/cloud/control/api/controller/`

| File | Role |
|------|------|
| `Open*Controller.java` | Route mappings (`OpenClusterController`, `OpenBackupController`, etc.) |
| `request/*.java` | Request DTOs — field names, types, validation annotations |

**Key differences from Milvus spec:**
- **Mixed HTTP methods**: GET (query params), POST/PUT/PATCH (requestBody), DELETE — not all-POST
- **Java validation framework**: Lombok `@Data`/`@Builder`/`@Jacksonized` + FluentValidate + Jakarta (`@NotBlank`, `@NotNull`, `@Size`, `@Pattern`)
- **Required field detection**: `@NotBlank`/`@NotNull` = required; `@FluentValidate({OptionalXxxValidator.class})` = optional with validation; bare field = optional
- **Inheritance**: some DTOs extend base classes via `@SuperBuilder` (e.g., `CreateDedicatedClusterRequest extends CreateClusterRequest`) — always check superclass for inherited fields
- **Pagination**: all List endpoints use `currentPage` (Integer, default 1) + `pageSize` (Integer, default 10) — query params, not body fields

**Cloud workflow** is similar to Milvus but with Java DTOs as source of truth:

1. **SCAN** — Read `Open*Controller.java` for routes, `request/*.java` for field names/types/annotations
2. **DIFF** — Compare against the relevant zdoc fragments; use `specs/openapi-cloud.json` only as an internal baseline/reference
3. **FIX** — Edit the relevant zdoc fragment under `zdoc/plugins/apifox-docs/meta/openapi/`
4. **VERIFY** — Parse touched JSON, confirm fields/examples in zdoc, then run zdoc generation checks when feasible
5. **BASELINE SYNC (optional)** — Only when explicitly requested, use `scripts/fix-openapi-cloud.js`, `scripts/edit-openapi.js --spec specs/openapi-cloud.json`, `verify-openapi-cloud.js`, and `merge-openapi.js` to update skill-local specs

## Syncing Cloud Source to Public Docs

The public documentation project (`zdoc/plugins/apifox-docs/meta/openapi/`) holds the split OpenAPI fragments that power the docs site and is the default target for REST doc edits. The skill's `specs/openapi-cloud.json` is only an internal baseline. When source code, the skill spec, or public fragments disagree, use the latest `zilliz-cloud` source as the API source of truth and preserve public-doc-only attributes in zdoc.

**Direction:** latest backend source → `zdoc/plugins/apifox-docs/meta/openapi/`; use `specs/openapi-cloud.json` as a comparison aid, not the final target.

**Workflow:**

1. **DIFF** — Compare paths and fields in `zilliz-cloud origin/master` against the project fragments (files 15–30)
2. **FLAG** — For each discrepancy, determine if the project is missing a real endpoint or if the project has an orphan
3. **APPROVE** — Show the action list to the user before making changes. Never unilaterally delete project endpoints; they may exist in the backend but not yet be scanned into the skill spec.
4. **EXECUTE** — Apply approved changes directly to the project JSON files:
   - Add missing paths from backend source or the confirmed baseline
   - Fix incorrect path names (e.g., plural vs singular)
   - Remove confirmed orphans (with user approval)
5. **PRUNE** — From `zdoc/`, run `npm run prune-apifox-specs` only when components may have changed. Apifox's per-tag exporter dumps the entire `components.schemas` block into every fragment regardless of usage; the script strips anything unreachable from `paths`. The plugin already tree-shakes at render time (`refGen.js` `resolveRefs`), so this is source hygiene only — no render impact. If it rewrites unrelated fragments without semantic removals, revert that churn and keep only the intended spec files.
6. **VERIFY** — Run `node plugins/apifox-docs/index.js fetch-apifox-docs -s plugins/apifox-docs/meta/openapi -o /private/tmp/zdoc-api-ref-test` to confirm the split specs merge and generate without rewriting checked-in reference pages. The `-s` folder argument is required in current zdoc; running without it only prints `Please provide specifications`.

**Key rules:**
- Latest backend source is canonical for API fields and behavior. zdoc fragments are canonical for public-doc authoring details (`x-target-lang`, `examples`, `x-include-target`, localized descriptions) and should be preserved.
- Path renames in the project that don't match the skill are usually incorrect (e.g., `createOnDemandClusters` vs `createOnDemandCluster`)
- `startByRemote` and `toExisting` are separate migration endpoints — both should exist in the project
- Skip import-related endpoints unless explicitly instructed otherwise
- When the target is explicitly the `master` branch of zdoc, check `git -C /Users/anthony/Documents/projects/zdoc branch --show-current` and switch before editing. If local `master` is ahead of `origin/master`, leave existing commits alone and apply the requested spec change on top.

## Public zdoc REST/OpenAPI Authoring

Use this workflow when editing `zdoc/plugins/apifox-docs/meta/openapi/*.json` directly for public REST docs or CLI-backed REST documentation.

Before authoring examples, scan the surrounding zdoc fragments as a corpus. Example naming, placeholder IDs, region choices, and success/error response shapes are intentionally consistent across files.

### Source Discovery

1. Fetch the latest backend source before judging whether a field exists:

```bash
git -C repos/zilliz-cloud fetch --all --prune
```

2. If the local checkout is sparse, add the directories needed for Cloud REST verification:

```bash
git -C repos/zilliz-cloud sparse-checkout add \
  vdc/global/cloud-control-api \
  vdc/global/cloud-service \
  vdc/cloud-client
```

3. Trace an endpoint through all relevant layers:
- Controllers and request DTOs: `vdc/global/cloud-control-api/src/main/java/com/zilliz/cloud/control/api/controller/`
- Service controllers for newer/alternate products: `vdc/global/cloud-service/src/main/java/com/zilliz/cloud/controller/`
- Frontend/API client wrappers: `vdc/cloud-client/packages/cloud/src/api/`

4. For On-Demand / Query Cluster APIs, do not rely only on `OpenClusterController`. Also scan VectorLake / Query Cluster paths:
- `vdc/global/cloud-service/src/main/java/com/zilliz/cloud/controller/vectorlake`
- `vdc/cloud-client/packages/cloud/src/api/queryCluster.ts`

### On-Demand / Query Cluster Mapping

The public On-Demand Cluster REST docs are backed by VectorLake Query Cluster APIs in `zilliz-cloud`. When reviewing Feishu drafts or adding public specs, map the internal source routes to public zdoc routes explicitly:

| Source route | Public route |
|--------------|--------------|
| `POST /cloud/v1/vectorlake/query-cluster` | `POST /v2/clusters/createOnDemandCluster` |
| `GET /cloud/v1/vectorlake/query-cluster` | `GET /v2/clusters/onDemandClusters` |
| `GET /cloud/v1/vectorlake/query-cluster/{clusterId}` | `GET /v2/clusters/onDemandClusters/{CLUSTER_ID}` |
| `DELETE /cloud/v1/vectorlake/query-cluster/{clusterId}` | `DELETE /v2/clusters/onDemandClusters/{CLUSTER_ID}` |
| `POST /cloud/v1/vectorlake/query-cluster/{clusterId}/modify` | `POST /v2/clusters/onDemandClusters/{CLUSTER_ID}/modify` |
| `POST /cloud/v1/vectorlake/query-cluster/{clusterId}/properties` | `POST /v2/clusters/onDemandClusters/{CLUSTER_ID}/properties` or the existing public update-info path, depending on the product-approved public URL |

Source files that were useful in practice:
- `VectorLakeController.java` — route mappings, project role requirements, request classes.
- `CreateQueryClusterCloudRequest.java` — create fields and validation. `cuSize` has `@JsonAlias("cu")`; docs should keep the approved public field name in the target spec.
- `ModifyQueryClusterAutoSuspendCloudRequest.java` — modify auto-suspend contract: required `projectId`, `regionId`, and `autoSuspend`.
- `ModifyQueryClusterPropertiesRequest.java` — update name/description contract: optional `newName`, optional `description`, at least one required.
- `vdc/cloud-client/packages/cloud/src/api/queryCluster.ts` and `schemas/queryCluster.ts` — frontend-facing naming and response normalization.

For `Modify On-Demand Cluster`, the backend accepts `autoSuspend` as a string with `<number><s|m|h>` format, such as `60s`, `5m`, or `1h`, and validates a minimum of 60 seconds. Do not model this endpoint as `PATCH`; the source and Feishu draft both use `POST /v2/clusters/onDemandClusters/{CLUSTER_ID}/modify`.

When reviewing Feishu API draft pages, use `lark-cli docs +fetch --api-version v2 --doc <url> --scope keyword --keyword "Modify On-Demand Cluster\|On-Demand Cluster\|onDemand" --doc-format markdown --format json` to extract the relevant section quickly, then verify every route and field against `zilliz-cloud origin/master`.

### Field Modeling Rules

- Add request fields where the create/update DTO accepts them.
- Add response fields where the describe/list response DTO or client model returns them.
- For list endpoints, add fields to the item schema, not only to the page wrapper.
- Detect required fields from Java annotations: `@NotBlank` and `@NotNull` are required; optional validators and bare fields are optional.
- Check inherited DTO fields (`extends`, `@SuperBuilder`) before concluding that a field is missing.
- Preserve existing property order and nearby schema style when adding fields.
- Use structured JSON edits and parse the file after changes; avoid string-only rewrites of OpenAPI JSON.

### Public Docs Attributes

Use `CUSTOM_ATTRIBUTES.md` in `zdoc/plugins/apifox-docs/` as the reference for supported public-doc extensions.

- `description`: write the English text on every new operation, parameter, schema, property, and example where the surrounding spec uses it.
- `x-i18n.zh-CN.description`: add the Chinese localized description beside every new English description.
- `x-i18n.zh-CN.summary`: add or preserve localized summaries for operations and tags.
- `x-include-target`: gate content by publication target (`zilliz` or `milvus`) only when the field, endpoint, tag, or example must be hidden from one target. Do not add it as decoration.
- `x-target-lang`: use on request/response/parameter examples to split English and Chinese examples (`en-US`, `zh-CN`). It is not a substitute for `x-i18n` field descriptions.
- `x-target-request` / `x-target-response`: use when examples correspond to a selected request or response tab.
- `x-base-urls` / `x-base-url-target`: use only for endpoints that need selectable base URLs, and keep target keys stable.
- Determine control-plane vs data-plane using `zdoc/plugins/apifox-docs/meta/plane-config.json`, not file numbering alone. `refGen.js` classifies pages with `getPlane(slug, target)`: if the generated slug contains any `controlPlaneKeywords[target]` keyword, the page is `control-plane`; otherwise it is `data-plane`.
- Control-plane `Authorization` parameters must say that the token is an API key with appropriate privileges. Do not mention cluster credentials, username/password, or `username:password` on control-plane endpoints. The only cluster-credential exception is Free, Serverless, and Dedicated cluster connection/data-plane access.

### Description Style

- State what the field or operation represents, not only its type.
- Mention constraints from source annotations when useful: length, enum values, format, defaults, and pagination behavior.
- Keep request field descriptions action-oriented: what the caller provides and how it is used.
- Keep response field descriptions observational: what the API returns and when it may be absent.
- For `description` fields specifically, describe the resource in plain product language, for example “A description of the project.” / “项目描述。” Do not imply it is required unless validation proves it.

### Examples

- Prefer OpenAPI `examples` objects over a single `example` when the surrounding operation already uses named examples.
- Match the local example-key style. Most zdoc fragments use numeric string keys (`"1"`, `"2"`, `"3"`) rather than semantic names; do not introduce names such as `Success` or `CreateProject` unless the file already uses semantic keys.
- Keep request examples minimal but complete: include all required fields and only optional fields needed to demonstrate the changed behavior.
- When adding a new parameter such as `description`, include it in create/update request examples and in describe/list response examples if the backend returns it.
- Keep example values realistic, reusable, and consistent with existing zdoc placeholders:
  - auth/header values: `Bearer {{TOKEN}}`, `application/json`
  - project IDs: `proj-xxxxxxxxxxxxxxxxxxxxxx`, `proj-xxx`, or an existing nearby `proj-...` shape
  - cluster IDs: `inxx-xxxxxxxxxxxxxxx`, `in01-xxxxxxxxxxxxxxx`; global cluster IDs: `glo-xxxxxxxxxxxxxxx`
  - job/backup/pipeline/integration IDs: `job-xxxxxxxxxxxxxxx`, `backupx-xxxxxxxxxxxxxx`, `pipe-xxx`, `integ-xxxxxxxxxxxxxxxxxxx`
  - collection/database/volume names: `my_collection`, `medium_articles`, `my_database`, `my_volume`
  - cluster/project names: `Dedicated-01`, `Free-01`, `Cluster-01`, `My Project`
  - regions: pair international and China examples when the endpoint is target/language sensitive, commonly `aws-us-west-2`, `aws-us-east-1`, `gcp-us-west1`, `ali-cn-hangzhou`, `ali-cn-beijing`
  - statuses/plans/types: use real enum casing such as `RUNNING`, `SERVING`, `Enterprise`, `Standard`, `Dedicated`, `Performance-optimized`
  - timestamps: use ISO UTC strings such as `2024-07-30T16:49:50Z`
  - descriptions: use short product-like values such as `Production vector search cluster`, `Production search workloads`, or `A pipeline that generates text embeddings and stores additional fields.`
- If there are language-specific examples, add both entries and mark them with `x-target-lang`.
- For Cloud v2 success responses, prefer `{"code":0,...}` unless the surrounding fragment uses legacy `200`; common auth/key failures are `{"code":1800,"message":"User hasn't authenticated"}` and `{"code":21119,"message":"The apikey is illegal."}`. Older v1 fragments may use `80002` token errors.
- When response examples map to `oneOf`/`anyOf` options, keep `x-target-response` values like `OPTION 1` / `OPTION 2`. When request examples map to tabbed request variants, keep `x-target-request`.
- Preserve existing `x-include-target`, `x-target-request`, `x-target-response`, and base-url filters on examples.

### Validation Checklist

Before finishing a zdoc REST/OpenAPI update:

1. Parse every touched JSON file with Node or `jq`.
2. Scan touched paths to confirm the new fields exist in request and response schemas where expected.
3. Confirm every added English `description` has matching `x-i18n.zh-CN.description`.
4. Confirm target filters were preserved and no `x-include-target` was added unnecessarily.
5. Confirm examples render the changed field in the right place and keep language/target filters.
6. If working in `zdoc`, run `npm run prune-apifox-specs` when components may have changed; if it rewrites unrelated fragments with no semantic removals, revert that unrelated churn.
7. Run `node plugins/apifox-docs/index.js fetch-apifox-docs -s plugins/apifox-docs/meta/openapi -o /private/tmp/zdoc-api-ref-test` when feasible to verify generated docs without modifying checked-in output.
