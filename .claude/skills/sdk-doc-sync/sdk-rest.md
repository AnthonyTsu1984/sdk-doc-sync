# RESTful API Reference (Milvus server)

**Source repo:** `repos/milvus` (sparse clone of `github.com/milvus-io/milvus` — only `internal/distributed/proxy/httpserver/` checked out)
**Latest release:** `v2.6.14` (as of 2026-04-14) — tags use plain `vX.Y.Z`
**Source files:** `internal/distributed/proxy/httpserver/`

| File | Role |
|------|------|
| `constant.go` | Route category prefixes (`/collections/`, `/entities/`, etc.) and action strings |
| `handler_v2.go` | All v2 route registrations via `RegisterRoutesToV2()` |
| `handler_v1.go` | v1 legacy routes |
| `request_v2.go` | Request struct definitions for v2 endpoints |

**Target files** (edited directly — no Feishu drive/bitable involved):

| File | Contents |
|------|----------|
| `specs/openapi-milvus.json` | Milvus server paths (`/v2/vectordb/*`, `/v1/vector/*`) + component schemas |
| `specs/openapi-cloud.json` | Zilliz Cloud paths (`/v1/clusters`, `/v2/clusters`, `/v2/alertRules`, etc.) |
| `specs/openapi.json` | Combined file — regenerated by merge script, **never edit directly** |

**Tag format:** `v*` (main milvus server releases, e.g., `v2.6.3`) — different from Go SDK `client/v*`.

**URL structure:** All v2 routes follow `/v2/vectordb/{category}/{action}`. All routes are `POST`.

## Workflow

**Phase 1: SCAN** — Read `handler_v2.go` for new/changed routes:

```bash
grep "router\.POST" repos/milvus/internal/distributed/proxy/httpserver/handler_v2.go
cat repos/milvus/internal/distributed/proxy/httpserver/constant.go
grep -A 20 "type FooReq struct" repos/milvus/internal/distributed/proxy/httpserver/request_v2.go
```

**Phase 2: DIFF** — Compare routes in source vs paths in `specs/openapi-milvus.json`.

**Phase 3: APPROVE** — Present new/changed/removed endpoints to user.

**Phase 4: EXECUTE** — Edit `specs/openapi-milvus.json` directly:
- Add new path objects following existing schema conventions
- Update request/response bodies to match new struct fields
- **Never edit `specs/openapi.json` directly** — regenerated by merge script

**Phase 5: MERGE:**

```bash
node .claude/skills/sdk-doc-sync/scripts/merge-openapi.js
```

## OpenAPI Path Object Conventions

```json
"/v2/vectordb/collections/create": {
  "post": {
    "summary": "Creates a collection",
    "description": "...",
    "tags": ["Collection Operations (V2)"],
    "parameters": [{ "$ref": "#/components/parameters/AuthorizationHeader" }],
    "requestBody": {
      "content": {
        "application/json": {
          "schema": { "$ref": "#/components/schemas/CreateCollectionRequest" }
        }
      }
    },
    "responses": { "200": { "description": "...", "content": { "..." } } },
    "x-i18n": { "zh-CN": { "summary": "..." } }
  }
}
```

Key conventions:
- Always include `"$ref": "#/components/parameters/AuthorizationHeader"` in `parameters`
- Tag names follow pattern `{Category} Operations (V2)` — check existing tags before adding new ones
- Include `x-i18n` with `zh-CN` translations for `summary` and field descriptions
- Cloud paths (`/v2/clusters`, `/v2/alertRules`, etc.) are in `openapi-cloud.json` — never mix

## Zilliz Cloud API (openapi-cloud.json)

**Source repo:** `repos/zilliz-cloud` (Java, Spring Boot)
**Source files:** `vdc/global/cloud-control-api/src/main/java/com/zilliz/cloud/control/api/controller/`

| File | Role |
|------|------|
| `Open*Controller.java` | Route mappings (`OpenClusterController`, `OpenBackupController`, etc.) |
| `request/*.java` | Request DTOs — field names, types, validation annotations |

**Key differences from Milvus spec:**
- **Mixed HTTP methods**: GET (query params), POST/PUT/PATCH (requestBody), DELETE — not all-POST
- **Java validation framework**: Lombok `@Data`/`@Builder`/`@Jacksonized` + FluentValidate + Jakarta (`@NotBlank`, `@NotNull`, `@Size`, `@Pattern`)
- **Required field detection**: `@NotBlank`/`@NotNull` = required; `@FluentValidate({OptionalXxxValidator.class})` = optional with validation; bare field = optional
- **Inheritance**: some DTOs extend base classes via `@SuperBuilder` (e.g., `CreateDedicatedClusterRequest extends CreateClusterRequest`) — always check superclass for inherited fields
- **Pagination**: all List endpoints use `currentPage` (Integer, default 1) + `pageSize` (Integer, default 10) — query params, not body fields

**Cloud workflow** is similar to Milvus but with Java DTOs as source of truth:

1. **SCAN** — Read `Open*Controller.java` for routes, `request/*.java` for field names/types/annotations
2. **DIFF** — Compare against `specs/openapi-cloud.json`
3. **FIX** — Use `scripts/fix-openapi-cloud.js` for batch fixes, or `scripts/edit-openapi.js --spec specs/openapi-cloud.json` for individual edits
4. **VERIFY** — Run `node scripts/verify-openapi-cloud.js` to compare spec vs Java DTOs
5. **MERGE** — Auto-called by fix scripts; or manual `node scripts/merge-openapi.js`

## Syncing Cloud Spec to Public Docs

The skill's `specs/openapi-cloud.json` is the **canonical source of truth** for the Zilliz Cloud REST API. The public documentation project (`zdoc/plugins/apifox-docs/meta/openapi/`) holds the split OpenAPI fragments that power the docs site. When the canonical spec changes, the project fragments must be updated to match.

**Direction:** `specs/openapi-cloud.json` → `zdoc/plugins/apifox-docs/meta/openapi/`

**Workflow:**

1. **DIFF** — Compare paths in the skill spec against the project fragments (files 15–28)
2. **FLAG** — For each discrepancy, determine if the project is missing a real endpoint or if the project has an orphan
3. **APPROVE** — Show the action list to the user before making changes. Never unilaterally delete project endpoints; they may exist in the backend but not yet be scanned into the skill spec.
4. **EXECUTE** — Apply approved changes directly to the project JSON files:
   - Add missing paths from the skill spec
   - Fix incorrect path names (e.g., plural vs singular)
   - Remove confirmed orphans (with user approval)
5. **PRUNE** — From `zdoc/`, run `npm run prune-apifox-specs`. Apifox's per-tag exporter dumps the entire `components.schemas` block into every fragment regardless of usage; the script strips anything unreachable from `paths`. The plugin already tree-shakes at render time (`refGen.js` `resolveRefs`), so this is source hygiene only — no render impact.
6. **VERIFY** — Run `node plugins/apifox-docs/index.js fetch-apifox-docs` to confirm the merged specs generate without errors

**Key rules:**
- The skill spec is canonical for the API definition, but the project may contain documentation-quality improvements (`x-target-lang`, `example`→`examples`, `x-include-target`) that should be preserved
- Path renames in the project that don't match the skill are usually incorrect (e.g., `createOnDemandClusters` vs `createOnDemandCluster`)
- `startByRemote` and `toExisting` are separate migration endpoints — both should exist in the project
- Skip import-related endpoints unless explicitly instructed otherwise
